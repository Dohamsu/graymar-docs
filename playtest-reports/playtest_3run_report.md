### NEW-1 (P0): IntentParserV2 "꺼내" 키워드 과매칭

**심각도**: 치명 (P0)
**발생 위치**: `server/src/engine/hub/intent-parser-v2.service.ts`

**현상**

INVESTIGATE ActionType의 키워드 목록에 "꺼내" 또는 "꺼내본"이 포함되어 있어, "물건을 꺼내 살피다"뿐 아니라 다음 모든 경우를 조사 행위로 오분류한다.

- "검을 꺼내들며" → INVESTIGATE (기대: THREATEN 또는 FIGHT)
- "금화를 꺼내" → INVESTIGATE (기대: BRIBE)
- "칼을 꺼내 위협하다" → INVESTIGATE (기대: THREATEN)

**근본 원인**

키워드 빈도 기반 매칭에서 "꺼내"가 범용 동사("take out")로 기능하여 어떤 행위든 1회 이상 매칭된다. 경쟁 키워드(THREATEN의 "으름장", BRIBE의 "금화 흔든다")가 0~1회에 그칠 때 INVESTIGATE가 승리한다.

**영향**

테스트 11턴 중 3턴 오분류. 특히 BRIBE와 THREATEN이 가장 빈번히 피해를 받는다. 스탯 기반 판정(THREATEN→ATK, BRIBE→SPEED)이 실제 플레이어 의도와 무관하게 결정되어 캐릭터 빌드 의미 훼손.

**수정 방향**

1. INVESTIGATE 키워드에서 "꺼내" 단독 항목 제거
2. 조사 문맥으로 한정: "꺼내서 살펴", "꺼내서 확인", "꺼내어 조사" 등으로 교체
3. 또는 "꺼내" + INVESTIGATE 다른 키워드 동시 매칭 시에만 유효 처리 (AND 조건)

### NEW-2 (P1): BRIBE 키워드 "주머니에서 꺼" 연속 매칭 실패

**심각도**: 높음 (P1)
**발생 위치**: `server/src/engine/hub/intent-parser-v2.service.ts` — BRIBE 키워드 정의

**현상**

BRIBE 키워드 "주머니에서 꺼"는 연속 문자열 매칭이므로, "호주머니에서 **금화를** 꺼내"처럼 중간에 목적어가 삽입되면 불일치한다.

```
"주머니에서 꺼" (키워드)
"호주머니에서 금화를 꺼내" (입력)
                 ↑ "금화를" 삽입으로 불일치
```

**근본 원인**

자연어 문장에서 "호주머니에서 X를 꺼내다" 패턴은 X에 해당하는 목적어가 필수적으로 삽입된다. 연속 매칭 방식은 이를 처리하지 못한다.

**영향**

BRIBE 행위가 거의 인식되지 않는다. SMUGGLER 프리셋의 핵심 플레이스타일(뇌물, 설득)이 사실상 비활성화 상태.

**수정 방향**

BRIBE 키워드 목록에 다음 항목 추가:

```
"금화를 꺼", "금화를 흔", "금화를 내밀", "금화 내밀",
"금화를 보여", "돈을 꺼", "돈을 내밀", "돈을 흔",
"은화를 꺼", "은화를 내밀", "뇌물", "뒷돈"
```

### NEW-3 (P1): 고집(Insistence) 카운터 미증가

**심각도**: 높음 (P1)
**발생 위치**: `server/src/turns/turns.service.ts` (추정), `server/src/engine/hub/intent-parser-v2.service.ts`

**현상**

Run 3 전 턴에서 `insistenceCount`가 0으로 유지됨. THREATEN을 의도한 입력을 3회 이상 반복했음에도 에스컬레이션이 발생하지 않았다.

**연쇄 원인 분석**

```
원인 A (파싱 오류 연쇄):
  T4: THREATEN → 정상 파싱, suppressedActionType 없음, count=0
  T5: THREATEN 의도 → INVESTIGATE 오분류 → suppressedActionType=THREATEN 기록 안 됨
  T6: FIGHT → 정상 파싱, count 초기화
  T7: BRIBE 의도 → INVESTIGATE 오분류 → 연속 카운트 불가

  결론: NEW-1 파싱 오류가 연쇄적으로 고집 카운터를 방해함

원인 B (독립적 가능성):
  turns.service.ts 내 actionHistory 누적 로직이 올바르게 동작하는지 독립 검증 필요
  insistenceCount가 ServerResultV1 → 클라이언트로 올바르게 전달되는지 확인 필요
```

**영향**

고집 에스컬레이션 시스템 사실상 미작동. 같은 LOCATION에서 반복적인 강한 행동 시도가 무시되며, "행동 그대로 실행" 서술 지시가 LLM에 전달되지 않는다.

**수정 방향**

1. NEW-1 파싱 오류 수정 후 재테스트
2. `turns.service.ts`에서 `actionHistory` 배열 누적 → `suppressedActionType` 감지 → `insistenceCount` 증가 로직 단계별 로그 추가로 검증
3. `insistenceCount >= 3` 조건 달성 시 LLM 프롬프트에 `escalated: true` 플래그가 실제로 포함되는지 확인

### 종합 평가 요약

3개 런 테스트를 통해 시스템의 핵심 파이프라인(HUB 순환, 이벤트 매칭, ResolveService, LLM 내러티브)은 전반적으로 안정적으로 동작하는 것이 확인되었다.

그러나 두 가지 치명적 문제가 게임플레이 경험의 핵심 요소를 훼손한다.

**첫째**, IntentParserV2의 "꺼내" 키워드 과매칭은 플레이어의 명확한 의도(위협, 뇌물)를 잘못된 ActionType(INVESTIGATE)으로 분류하여 캐릭터 빌드가 판정에 반영되지 않는 문제를 유발한다. 이는 단순한 UX 문제가 아니라 게임의 핵심 설계 원칙인 "캐릭터 스탯이 확률에 직접 영향"을 훼손한다.

**둘째**, 고집(Insistence) 에스컬레이션 시스템이 전혀 동작하지 않아 CLAUDE.md에 명시된 핵심 설계 목표(같은 LOCATION 3회 연속 → 강한 actionType 승격)가 플레이어에게 전혀 체감되지 않는다.

이 두 문제는 서로 연쇄하여 서로를 악화시킨다. 파싱 오류가 고집 카운터를 방해하고, 고집 시스템의 미작동이 파싱 오류의 실제 영향을 더 크게 만든다.
