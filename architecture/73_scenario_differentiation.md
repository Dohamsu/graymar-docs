## 1. 문제 — "소재가 다른데 느낌이 같다"

플레이 몇 턴이면 서로 다른 시나리오가 동형(isomorphic)으로 수렴한다. 소재(항구 부패 vs 극야 실종)·이름·이미지·톤은 다르지만, **진행의 뼈대**가 동일하다.

### 1.1 체감의 층위 (검토 v2 — 원인을 시간축으로 분해)

"비슷하다"는 체감은 단일 원인이 아니라 플레이 구간별로 다른 원인이 만든다:

| 구간 | 체감을 가르는 것 | 실측 근거 |
|------|------------------|-----------|
| **첫 5분** | 프리셋 **명명·원형 대칭** + 프롤로그 구조(여관/선술집에서 의뢰인이 다가와 {HOOK}) | §2.3 |
| **중반 (LOCATION 루프)** | 이벤트 **밀도** (123 vs 36 — 별빛모래는 폴백·자유대화 잦음) + 타입 분포 | §2.2 |
| **장기 (아크~엔딩)** | ArcRoute 3-enum 고정 삼각 | §2.4 |

→ §2.4를 "단일 최대 원인"으로 꼽았으나, 층위로 보면 **첫인상은 프리셋, 항속 체감은 이벤트 밀도, 종반 수렴은 아크**가 각각 지배한다. 개선 순서도 이 층위를 따른다 (§5).

## 2. 진단 — 비슷함의 3층 원인 (실측)

### 2.1 엔진층 — 플로우가 구조적으로 강제됨 (팩이 못 바꿈)

서버가 모든 로직을 결정론 처리하고 LLM은 서술만 한다(핵심 불변식 1·2). 그래서 **모든 시나리오가 동일 루프**를 돈다:

```
HUB 장소 선택 → LOCATION 진입 → 행동 입력 → 1d6 + floor(stat/4) + baseMod 판정
  → SUCCESS(≥5)/PARTIAL(3~4)/FAIL(<3) → 이벤트 매칭 → LLM 서술 → 다시 HUB
```

여기에 NPC 소개(별칭→실명), fact 점진 공개, posture 5종, 대화 잠금, 아크 커밋, 안정도별 엔딩까지 **엔진이 고정**한다. 팩은 이 루프를 바꿀 수 없다. 이건 결함이 아니라 플랫폼 자산(버그 적음·팩 추가가 쌈).

### 2.2 콘텐츠층 — 두 팩이 같은 "틀"을 채움

| 요소 | 그레이마르 | 별빛모래 | 동일도 |
|------|-----------|----------|--------|
| 퀘스트 단계 | S0_ARRIVE→S5_RESOLVE (6) | S0_ARRIVE→S5_RESOLVE (6) | 골격 동일 (팩 계약 — 불변식 45) |
| **아크 루트** | EXPOSE_CORRUPTION / PROFIT_FROM_CHAOS / ALLY_GUARD | **같은 3 enum** (엔딩 제목만 "귀환한 이름들/새 닻/잠긴 꿈") | **완전 동일** (endings.json 실측) |
| NPC 계층 | CORE 6 / SUB 12 / BG 25 | CORE 6 / SUB 12 / BG 30 | 거의 동일 |
| 장소 | 7 | 8 | 유사 |
| 인시던트 kind | CRIMINAL/POLITICAL/SOCIAL/ECONOMIC/MILITARY | 같은 5종 | 동일 |
| 이벤트 type | RUMOR/ENCOUNTER/OPPORTUNITY/ARC_HINT/AMBUSH/FACTION… | 같은 taxonomy | 동일 |
| 이벤트 수 | 123 | 36 | **밀도 3.4배 차** (별빛모래 폴백 잦음) |

**silverdeen_v1은 12개** (검토 v2 추가) — 밀도 문제의 극단 사례. 미니 검증 팩이라 의도된 것이지만, "팩 최소 이벤트 수" 기준이 팩 계약에 없다는 증거이기도 하다 → A2에 팩 계약 확장 포함.

### 2.3 프리셋 — "1:1 복제"의 실체는 명명·구조 대칭 (검토 v2 정정)

첫 5분 체감을 결정하는 지점. **단, 초안의 "거의 1:1 복제"는 과장이었다** — 스탯 실측 대조 결과:

| 그레이마르 | 별빛모래 | 접두/원형 | **스탯 실측** |
|-----------|----------|-----------|---------------|
| **부두** 노동자 | **부두** 하역꾼 | 탱커 (접두 동일) | **완전 동일** (str14/dex4/wit4/con14/per5/cha5 — 6개 스탯 byte-identical, 실측) |
| **밀수**업자 | **밀수** 잠입꾼 | 잠입 (접두 동일) | 상이 (cha 12→5, per 9→12 — 사교→관찰형) |
| 약초상 | 오염 치유사 | 서포트/치유 | 상이 (wit/per 하향, HP 상향) |
| **몰락** 귀족 | **몰락** 관측생 | 접두 동일 | **정반대** (cha 16→6, wit/per 특화 학자형) |
| **떠돌이** 검투사 | **떠돌이** 순례자 | 접두 동일 | **정반대** (str 14→5, cha 6→14 사교형) |
| 탈영병 | 귀환 생존자 | 과거를 진 자 | 유사 골격, 분배 상이 |

**정정된 진단**: 메카닉(스탯 원형)은 이미 절반 이상 차별화되어 있다. 실제 문제는 ① **명명 접두 복제** (부두/밀수/몰락/떠돌이 — 목록 화면에서 즉시 데자뷰), ② **6종 고정 수의 대칭 구조**, ③ **배경 서사의 원형 재탕** ("~을 목격하고 쫓겨난", "조직 와해 후"). → A1의 스코프는 "전면 재편"이 아니라 **명명·서사·수 변화**로 좁힌다 (§4 A1).

### 2.4 아크 수렴 — Canonical ArcRoute enum

`ArcRoute`(`EXPOSE_CORRUPTION / PROFIT_FROM_CHAOS / ALLY_GUARD`)는 엔진 고정 enum이며 **정확히 5개 서비스 파일이 enum 값을 분기**한다(실측: `arc-state.ts`, `ending.ts`, `ending-generator.service.ts`, `summary-builder.service.ts`, `arc.service.ts`). 따라서 **모든 시나리오의 최종 선택이 "폭로/이익추구/동맹수호" 삼각에 강제 매핑**된다. 별빛모래도 엔딩 제목·에필로그만 팩화됐을 뿐(endings.json arcRouteEndings) 결정 아키텍처는 동일하다.

**검토 v2 — 결합 폭 정밀화**:
- enum 값 분기: 서버 5파일 + **클라 2파일** (`types/game.ts`, `QuestTab.tsx` — 초안 미언급).
- `routeTag`/`arcRouteTag` 문자열: 10파일에 흐르지만 대부분 패스스루. 불변식 18(Procedural Plot Protection — 동적 이벤트 arcRouteTag 금지)과 routeCommitChoices(arch/68 부록 F)가 이 문자열 계약 위에 있다.
- **표현층은 이미 콘텐츠**: 루트별 제목·에필로그·keyAllies는 endings.json이 팩별 보유. 엔진에 남은 표현 결합은 `defaultArcTitle` 폴백 맵 정도.

→ 결과: **도착 → 단서 조사 → NPC 접촉 → fact 발견 → S0~S5 진행 → 3개 중 커밋 → 안정도별 엔딩** 궤적이 소재와 무관하게 반복된다.

## 3. 설계 원칙

> 엔진 루프는 유지(자산)하되, **각 팩에 "정체성 축" 하나씩**을 주고 **콘텐츠 템플릿 대칭을 의도적으로 깬다.** 완전 분기는 팩마다 엔지니어링·테스트 비용이 드니, **최소 엔진 비용으로 체감을 최대한 벌리는** 순서로 간다. 목표는 "팩마다 다시 만들기"가 아니라 "얇은 훅 + 콘텐츠 탈템플릿".

추가 원칙 (검토 v2):
- **기존 인프라 재사용 우선** — 새 시스템 신설 전에 연쇄 조건(Layer 2)·traitEffects 훅·EventMatcher weight 등 기존 자리에 얹을 수 있는지 먼저 본다.
- **추가형(additive) > 일반화(refactor)** — 널리 참조되는 기존 개념(Heat 등)을 리팩터링하지 않고 옆에 새 축을 추가한다.
- **계측 없는 차별화 없음** — "체감이 달라졌다"는 §8 지표로 확인한다.

## 4. 개선 방향 (Tier별)

### Tier A — 콘텐츠만 (엔진 0줄, 즉시 가능) · 임팩트 중~상

- **A1. 프리셋 탈템플릿화** — 스코프 정정(§2.3): 스탯 원형은 이미 상당히 다르므로, ① 명명 접두 복제 폐기(부두/밀수/몰락/떠돌이), ② 팩별 프리셋 **수 변화**(4~7 — 6종 대칭 해체), ③ 배경 서사 원형 재편(별빛모래는 "꿈" 원형 — 몽유병자·닻지기·기록수녀 등). protagonistTheme/prologueHook도 함께 재작성.
  - ⚠️ **순서 리스크 (검토 v2)**: star_sand 프리셋 초상화 12장(6종×2성별, `public/preset-portraits/star_sand_v1/`)이 이미 배치됨. A1로 원형을 갈아엎으면 폐기된다. **원형·명명 확정 → 초상화 재생성** 순서를 고정하고, 비용에 이미지 재생성(수 변화 시 ±)을 포함할 것. creation-bundle(arch/71)이 프리셋을 완전 콘텐츠화했으므로 코드 0줄은 유효 — 단 클라 `PACK_PRESET_PORTRAITS` 매핑 갱신 1곳은 따라간다.
- **A2. 이벤트 밀도·분포 차별화** — 팩 성격에 맞춰 타입 가중치를 분리: 그레이마르=FACTION/AMBUSH(정치 스릴러), 별빛모래=RUMOR/ARC_HINT(미스터리). 별빛모래 이벤트 수 상향(36→80+)으로 자유대화 폴백 감소. EventMatcher weight가 이미 콘텐츠 제어라 순수 저작 작업.
  - **팩 계약 확장 (검토 v2)**: "장소당 최소 이벤트 수" 기준을 팩 계약(불변식 45 문단)에 추가하고 ContentValidator 경고로 배선 — silverdeen(12개) 같은 저밀도 팩이 조용히 폴백 지옥이 되는 것을 구조적으로 방지.
- **A3. 아크 서사 프레이밍 심화** — arcRouteEndings·arcHint·keyAllies를 밀어 enum이 같아도 "고유 딜레마"로 읽히게. 엔딩 표현층 팩화는 선행 착수 완료(endings.json 팩화 + defeatLine 외부화 — server `700aefb`).
- **A4. 톤·문체·감각 팔레트 고정** — **절반 기구현 (검토 v2)**: 시스템 프롬프트 세계관 주입(`world.settingLine`/`regionSummary`)은 arch/63 ④에서 완료, scene_shells/atmospheres도 팩 콘텐츠. **신규는 "감각 팔레트 positive 풀"** — 팩별 감각 어휘 묶음(극야: 오로라/고래기름/성에/침묵/젖은 모피)을 scenario.json에 선언하고 프롬프트에 "다음 중 선택" 형태로 주입. LLM 설계 원칙 #2(negative보다 positive 풀)와 정합 — 금지어 규칙보다 준수율이 높다.

→ **A2+A1이 가성비 최고.** 코드 0줄(A2 검증 배선 제외)로 "도착 직후 동형 수렴"(A1)과 "중반 루프 데자뷰"(A2)를 동시 완화.

### Tier B — 경량 엔진 훅 (팩이 켜는 축) · 임팩트 상

- **B1. ★ 팩 전용 "세계 축" 게이지 — 추가형 packMeters (검토 v2 설계 교체)**
  - 초안의 "Heat/pressure를 명명된 팩 미터로 일반화"는 **기각** — `hubHeat`는 27개 파일이 참조(프롬프트·이벤트 매칭·시그널·NPC 반응·클라 HUD)해서 일반화 리팩터링 비용이 임팩트를 잠식한다.
  - **교체 설계 — 추가형**: Heat는 그대로 두고 옆에 팩 선언 미터를 추가한다.
    ```
    scenario.json (팩 계약 확장):
    "meters": [{
      "id": "DREAM_TAINT", "name": "꿈 오염", "initial": 0,
      "perTurnDelta": 1,                    // WorldTick 훅에서 적용 (시간대/장소 조건부 허용)
      "maxDeltaPerTurn": 8,                 // 불변식 9(Heat ±8 clamp)에 준하는 규약
      "thresholds": [
        { "at": 70, "conditionId": "DREAM_SURGE" },   // 기존 연쇄 조건 시스템에 등록
        { "at": 100, "endingTrigger": "BAD_DREAM" }    // C2 연계 (선택)
      ]
    }]
    ```
  - **핵심 절감 — 기존 인프라 재사용**: 임계 발동은 신규 시스템이 아니라 **연쇄 반응 시스템(Layer 2)의 `activeConditions`/`triggeredConditions` 파이프에 팩 조건을 등록**하는 것. 장소 봉쇄·판정 보정(blockedActions −2)·시그널 피드 알림이 이미 그 파이프에 있다.
  - **선행 필요 작업**: 조건 표시 텍스트(`CONDITION_SIGNALS` 등)가 6개 파일에 하드코딩되어 있어(turns.service, context-builder, nano-event-director, situation-generator, consequence-processor, location-state) 팩 조건을 받으려면 **조건 정의 외부화**가 먼저다 — 불변식 45의 기존 부채 청산과 겹치는 작업.
  - 저장은 `worldState.packMeters: Record<string, number>` — 미터 미선언 팩(그레이마르)은 필드 자체가 비어 기존 동작 무변경.
  - 클라: HUB 상태줄에 미터 1개 표시(이름·수치·경고색) — 소규모.
  - **단일 최고 레버리지 유지** — "보기에 다른"이 아니라 "플레이가 다른" 팩. 추가형이라 A와 병행 가능한 규모로 내려온다.
- **B2. ArcRoute 팩 정의화 — 단계 축소 (검토 v2)**
  - 1단계: **루트 수는 3 고정, ID·라벨만 팩 선언** — 서버 5파일의 enum 분기 제거 + `defaultArcTitle` 폴백을 콘텐츠 참조로 + 클라 2파일(`game.ts`, `QuestTab.tsx`) 타입 완화. routeTag 문자열 10파일은 패스스루라 계약만 문서화.
  - 2단계(팩 3개+ 시점): 루트 **수** 가변 — 엔딩 그리드(4루트×3안정도=12분기)와 HUB 커밋 선택지(routeCommitChoices)가 수 가변을 전제로 재설계 필요. 비용 중상.
- **B3. 판정 축 팩 변주** — 1d6+stat 유지하되 팩 고유 위험(별빛모래: NIGHT 전역 페널티·꿈 오염 상태이상)을 판정에 주입.
  - **기존 훅 재사용 (검토 v2)**: `traitEffects.nightBonus/dayPenalty`(불변식 32)가 이미 resolve에 시간대 보정을 주입하는 자리다. 같은 자리에 팩 전역 `packModifiers`를 얹으면 되므로 신규 개척이 아니라 기존 자리 확장. 상태이상 신설(꿈 오염)은 Status ID가 엔진 enum(BLEED/POISON/STUN/WEAKEN/FORTIFY — specs/status_effect_system_v1 §10)이라 별도 확장 필요 — B1 임계 효과(판정 보정)로 우선 대체하고 상태이상 팩화는 후순위.

### Tier C — 구조적 (큰 작업) · 선택

- **C1. 매크로 구조 팩별화** — 팩마다 진행 골격(선형/방사형/시간압박형) 분리.
  - **전제 명시 (검토 v2)**: "그레이마르=24노드 DAG"는 현재 DAG가 **비프로덕션 모드**라는 사실을 전제로 재평가해야 한다 — 기본 런은 `hub` 모드이고 DAG는 `mode: 'dag'` 옵션 경로이며, star_sand는 graph.json 자체가 없다. 즉 C1은 "팩별 골격 선택"이 아니라 사실상 **DAG 모드의 프로덕션 승격 + 전 팩 graph 저작**이라는 별개 대형 과제.
- **C2. 승패·엔딩 조건 다변화** — S5+안정도 외 팩 고유 종료(꿈 오염 100% = 배드엔딩 등). B1의 `endingTrigger` 임계로 자연 연계.

## 10. 미해결 가정·비용 주의 (검토 v3 — 평가 반론 반영)

착수 전 반드시 인지할 가정·낙관 지점. 이 문서는 **코드 비용은 정직하나 저작·선행 비용과 착수 정당화가 낙관적**이라는 한 방향의 편향이 있다.

### 10.1 A2의 "즉시·코드 0줄"은 총비용이 아니다

"이벤트 36→80+"는 코드 0줄이 맞지만 **이 계획 전체에서 가장 큰 노동**이다 — 이벤트 ~50개 신작(payload/choices/discoverableFact/NPC배정/affordance/tags) + 밸런스 QA. Tier A의 "즉시 가능" 프레이밍은 **코드 비용과 총비용을 혼동**시킨다. A2는 "코드 0줄, 저작 대형"으로 재라벨하고, 착수 시 저작·플레이테스트 공수를 별도 산정할 것.

### 10.2 B1의 "조건 정의 외부화 선행"이 숨은 대형 과제

B1을 "A와 병행 가능한 규모"로 결론냈으나, 그 전제인 조건 정의 외부화(§4 B1 6파일 하드코딩 청산)는 **B1 본체보다 클 수 있다** — 사실상 불변식 45 조건 부채의 완결. 또한 Layer 2 재사용은 LOCKDOWN/RIOT 조건이 Heat/unrest 축에 종속돼 있어, 팩 조건이 condition→effect→signal→prompt 경로에 임의 의미를 주입하려면 **재사용이 아니라 추가 수술**이 될 여지가 있다(이 결합은 미검증). → B1은 "선행(조건 외부화) 규모 스파이크"를 먼저 뜬 뒤 병행 가능성을 재판정.

### 10.3 silverdeen 근거의 유통기한

§2.2/A2의 "최소 밀도 기준"이 silverdeen(12 이벤트)을 핵심 증거로 든다. **silverdeen은 폐기 예정**이므로 이 증거는 곧 소멸한다. 최소 밀도 논거는 **star_sand(36) 단독으로 다시 세울 것** — 36도 이미 폴백 과다를 유발한다는 실측(playtest)이 있으므로 논거 자체는 유지된다.

### 10.4 "비슷함"이 우선순위라는 정당화 부재 (최상위 가정)

전체 계획이 **단일 정성 관찰**(체감상 비슷함)에 근거한다. §8이 "'비슷하다' 언급 감소"를 지표로 넣었지만 **사용자 불만의 baseline이 없다.** 명시 가정: *유저 신호(리텐션/리뷰/버그리포트)가 확인되지 않으면 Tier B·C는 보류하고 A(저리스크 콘텐츠)만 진행한다.* B1 엔진 투자는 이 신호가 잡힌 뒤 착수하는 것이 안전하다.

### 10.5 실행 착수 전 보강 필요 (2건)

- **공수 sizing 부재** — 항목별 임팩트·순서는 있으나 t-shirt 규모(일수)가 없어 A2(대형 저작) vs B1(중형 엔지니어링) 실제 선택을 못 돕는다. 착수 문서화 시 항목별 대략 공수 추가.
- **A1 ROI 하락(순서 사고)** — star_sand 초상화 12장을 이미 생성·배선한 상태(`public/preset-portraits/star_sand_v1/`, `PACK_PRESET_PORTRAITS`)에서 A1이 프리셋 원형을 갈아엎으면 전량 매몰된다. §10.4 결론과 합치면 **A1은 A2보다 뒤로** 미루고, 원형 재편이 확정되기 전엔 추가 에셋 제작을 동결.

### 11.2 진단 정정 — A2 "증량 → 폴백 감소"는 부분 오진

문서 초안(§4 A2)은 "이벤트 36→80으로 증량하면 폴백 감소"라 봤다. **A2 증량은 완료됐으나(80개, 장소당 8~11개 균등, 미스터리 타입 RUMOR 19+ARC_HINT 11), 12턴에 매칭된 저작 이벤트는 여전히 2종뿐**이었다. 즉 개수는 필요조건이지 충분조건이 아니다. 코드 실측으로 좁힌 **진짜 병목 3가지**:

1. **affordance 커버리지 편중** — `LOC_SS_INN` 11개 이벤트 중 weight 상위 4개(pri5/w70)가 모두 INVESTIGATE/OBSERVE/PERSUADE 중심. **TALK affordance는 1개(SAME_WORDS), HELP는 0개.** 플레이어가 "도움을 준다(HELP)"·"말을 건다(TALK)"를 입력하면 상위 이벤트가 EventMatcher Step 4(affordance ∩ actionType) 필터에서 탈락 → 소수 후보로 수렴 → 최하위 weight 이벤트(LAMP_TREMOR w50)가 걸림.
2. **대화잠금 + NPC연속성 강선호** — 불변식 26 대화잠금(같은 NPC 최대 4턴) + `EventMatcher.calcNpcContinuityBonus`(+25) + `calcNpcSwitchPenalty`(대화계열 ×3 = -90)가 같은 NPC 이벤트를 강하게 유지. RIVA_HAUL 3턴 연속(T3~5)의 원인 — 이건 "장면 연속성" 의도 동작이나 **다양성 체감은 저하**.
3. **세계관 표면화 실패** — 중반 LOCATION 서사에 극야/오로라/별고래 고유명이 두 표본 다 0회. 프롤로그(이렌 의뢰)의 강한 고유성이 중반에 유지 안 됨 → "일반 항구 도시"로 읽힘.

### 11.8 A2' 후속 — 탐색 행동 WORLD_EVENT 승격 (2026-07-15)

§11.7이 남긴 후속 과제(저작 이벤트 매칭 빈도)를 파고든 결과, 근본은 **`determineTurnModeCore`가 저작 이벤트(EVT_)를 WORLD_EVENT 턴에만 매칭**하고, 그 조건이 첫 진입/pressure≥70/questFact로 좁다는 것. 그 외 플레이어 행동은 PLAYER_DIRECTED→`FREE_PLAYER`, 대화 연속→`FREE_CONV`로 처리되어 **저작 이벤트를 아예 안 탄다**. 토픽1에서 affordance를 보강해도 TurnMode가 문을 안 열어준 것.

**구현**: `determineTurnModeCore`에 조건 추가 — 대화 상대 없는 탐색 행동(INVESTIGATE/OBSERVE/SEARCH)이고 현재 장소에 그 행동으로 매칭 가능한 저작 이벤트가 있으면(`exploreEventAvailable`) PLAYER_DIRECTED 대신 **WORLD_EVENT로 승격**. 대화 연속(2)이 먼저 걸러지므로 대화 중 탐색은 대화 유지(회귀 없음). Player-First 정합(플레이어 탐색 행동에 맞는 저작 이벤트 제공). 단위 테스트 5종 추가(총 61 passed).

**검증 (통제 — 탐색 위주 플레이)**: playtest 랜덤 행동은 탐색 5/14(36%)뿐이고 저작 이벤트가 NPC를 세팅하면 후속 탐색이 대화 잠금(불변식 26)에 흡수되어 집계에 안 드러난다. 그래서 탐색 위주 통제 스크립트로 대조:

| 조건 | 저작 이벤트 종수 | 승격률 |
|------|-----------------|--------|
| 개선 전 (탐색=PLAYER_DIRECTED) | 장소 진입 시 1개씩만 (탐색은 FREE_PLAYER) | 0% |
| **개선 후 (탐색 위주 16스텝)** | **5종**(FIRST_DREAM/HELUN_MUTTER/EMPTY_COT/DREAM_MURMUR/LAMP_TREMOR) | **탐색 전부 WORLD_EVENT 승격** |

여관 한 곳에서 매 탐색이 다른 저작 이벤트를 탐(cross-visit cooldown + 방문 내 하드캡이 다양성 보장). **토픽1(affordance)+A2' 후속(TurnMode 승격)이 합쳐져야 실효** — affordance는 후보를, 승격은 매칭 기회를 연다.

**한계·판정**: 개선은 "대화 상대 없는 탐색"에 국소 작동. 저작 이벤트가 NPC를 세팅하면 후속은 대화 잠금에 흡수되므로, **대화·전투·거래 위주 플레이에선 여전히 FREE 우세**. 저작 이벤트 빈도의 추가 상승은 대화 잠금 정책 완화(고위험)를 건드려야 하므로 **여기까지가 저리스크 경계**. 실유저의 탐색 비중이 확인되면(제품 신호) 충분, 아니면 대화 잠금 완화는 보류.
