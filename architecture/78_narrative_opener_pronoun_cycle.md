## 1. 범위와 결정 이력 (그릴링 합의)

핵심 3건만: ① 즉흥 별칭 억제(재현 계측 선행) ② 개시어 임계 3→2 ③ 대명사 계열 합산.
소액 백로그('~군요' 표류·P5 죽은 규칙·D4-3b)는 제외.

**그릴링 중 사실 교정 2건**:
- 인계 메모의 즉흥 별칭 예시("입이 가벼운 술꾼")는 **NPC_RENNICK의 콘텐츠 unknownAlias**였다 (shortAlias "술꾼" 보유, deduplicateAliases 정상 커버리지 안). 메모 예시 부정확.
- 실제 관찰(V9 flaky "책임자는")은 **콘텐츠 별칭의 축약형** — 마이렐 unknownAlias "야간 경비 책임자"의 부분 문자열. 즉흥 창작이 아니라 **별칭 축약 반복**이라는 별개 클래스 (§5 백로그).

## 2. 구현

### D5 계측 센서 (scripts/playtest.py — D4 계측군 확장, 게이트 아님)

- **[D5-1] 대명사 개시어율** — 서버 `extractOverusedOpeners`와 동일 전처리(마커 대사·인용 제거, `[.!?…다]` 문장 분리, 첫 단어 한글 2+자) 재현. 대명사 화이트리스트 계열을 1키로 합산한 문장 비율. 패치 전/후 비교 기준 지표.
- **[D5-2] 지칭 명사구 반복** — 주어형 stem(은/는/이/가 조사 앞 한글 2~6자) 런 전체 누적 3회+. 분류: `CONTENT_ALIAS`(V9-c `_npc_alias_pool` 매칭) / `NON_ALIAS`(즉흥 지칭·일반 명사 후보 — 사람 지칭 여부는 리포트 검토로 판정). 용언 어간('떨리'·'들려오') 오탐은 후보 표면화 목적상 허용.

### 서버 패치 (context-builder.service.ts `extractOverusedOpeners`)

1. **임계 3→2** (`OPENER_REPEAT_MIN`) — 반복이 굳기 전 조기 개입. 6턴 창·top 3·"최대 1개" 지시문은 유지.
2. **대명사 화이트리스트 합산** (`PRONOUN_OPENERS` — 그는/그가/그의/그를/그도/그에게 + 그녀 동형 12종): '그는' 2회 + '그가' 1회처럼 형태 분산으로 개별 임계를 새던 구멍을 1키 합산으로 차단. **명시 목록만** — 그러나/그리고/그때/그런 접두 오탐 억제 금지("한복"→"한복판" 오매칭 계열 위험).
3. 프롬프트에는 실관측 최빈 2형태만 병기 노출(예: `"그는/그가"`), **동률 시 대명사 키 우선** 타이브레이크(표적이 top 3 슬롯 경쟁에서 밀리지 않게).

유닛: `context-builder.overused-openers.spec.ts` 7케이스 (임계 2·합산·남/녀 통합·비대명사 비합산·대사/마커 제외·top 3 경쟁). 전체 1,397 passed.

- **판정**: 합의 기준(3런 평균 상대 -30%) **미달** — 부분 달성. 회귀 0.

### 왜 -30%에 못 미쳤나 (분석) — ⚠️ arch/79 P1에서 해석 정정됨

> **정정 (2026-07-18, arch/79 §6)**: 아래 "soft 준수 천장" 해석은 회고 상관 1,556턴에서 수정됐다. 잔여 대명사율의 실체는 **프롬프트 12k+ 비대 턴** — <10k 턴에서는 전 턴 밴드에서 3~5%로 지시가 잘 먹힌다(층화 검증). 근본 처방은 지시 강화가 아니라 프롬프트 총량 예산(arch/79 P3-1).

패치 후 대명사 합산 키는 사실상 **매 창에서 활성**(6턴 창에 2회+는 자명)인데도 잔여 ~15~16%가 남는다. 즉 지시문("해당 개시 문장 최대 1개")의 **soft 준수 천장**이 원인이며(LLM 설계 원칙 4 — soft 지시 자주 무시), 감지 측 추가 강화(창 확대·임계 하향)는 대명사 축에는 **무효**다 — 이미 항상 감지되고 있기 때문. 감지 창 확대(그릴링에서 2차 후보로 합의)는 비대명사 장주기 반복에만 유효한 별개 레버.

## 4. 즉흥 지칭 착수 트리거 — 미충족 (백로그 강등)

합의 트리거: baseline 2런 중 1런+에서 동일 즉흥 '사람 지칭' 명사구(NON_ALIAS) 3회+ 반복.

- baseline 1: NON_ALIAS = 공기5·떨리4·냄새4·들려오4·안개3·시선3·눈빛3 — **전부 환경 명사·용언 어간, 사람 지칭 0건**
- baseline 2: NON_ALIAS = 공기5·떨리3·냄새3·햇살3·행인3 — 행인은 사람이나 특정 지칭 아닌 집합 명사. **트리거 미충족**
- 사람 지칭성 반복(책임자5·회계사·실무자·노부인·사내)은 **전부 CONTENT_ALIAS 매칭** — 즉흥 창작이 아니라 콘텐츠 별칭(또는 그 축약형) 사용

**결론**: "LLM 즉흥 별칭 반복"이라는 가설 자체가 실측으로 기각. 센서([D5-2])는 상시 유지하여 재발 감시.

## 5. 잔여·백로그 (이번 사이클 밖 관찰)

1. **대명사 개시 2차 처방 (소유자 결정 대기)** — 감지는 포화 상태이므로 남은 레버는 ① 지시문 강화(positive 재작성: "이번 턴 문장 개시는 행동·소리·공간 묘사로" 수준을 디렉티브로 승격) ② 후처리(문장 개시 대명사 치환 — 문장 파괴 위험, 최후 수단) ③ 현 수준(-20%) 수용. chatty 짝이 -30%를 충족했고 brawler baseline이 원래 낮았던 점을 고려하면 표본 누적 후 재판정도 선택지.
2. **별칭 축약형 반복** (신규 클래스) — "야간 경비 책임자"→"책임자"×5 등 콘텐츠 별칭의 축약형은 deduplicateAliases(완전 일치 기반)가 모른다. 억제하려면 별칭 축약형 파생 매칭 필요 — 단 "책임자"는 일반 명사이기도 해 오치환 위험, 신중 설계 필요.
3. **관형형 수식어 반복** — V9 '섞인' 5~7회(3턴 내), '무슨' 5회. 개시어(문두)도 별칭도 아닌 문중 반복으로 기존 채널(bigram·opener) 사각. arch/77 V9 flaky 판정과 동일 클래스.
4. 비대명사 장주기 반복이 재관측되면 감지 창 확대(런 누적)가 유효한 레버 (대명사 축에는 무효 — §3 분석).

## 7. 2차 처방 실행 — [서술 지칭 규칙] 디렉티브 (2026-07-19, server b800088)

§5-1 백로그의 ①안(디렉티브 승격) 실행. arch/79 완료 후에도 대화 턴 대명사 기저(29~35%)가 크기와 무관하게 잔존함이 확정된 데 따른 처방.

**설계** (3반복 튜닝 — 게이트 A/B 실측 기반):
1. **v1 감지 의존 발화**: 세션 창 임계 도달 시만 발화 → 15턴 중 6턴 커버. 런 내 A/B: ON 25.6% vs OFF 대화 42.9% — **디렉티브는 먹히나 커버리지 구멍**.
2. **v2 대화 턴 상시 발화**: 커버리지 확장 → 대화 턴 10.0% 달성. 그러나 **풍선효과 실측** (LLM 원칙 5): 대명사 대신 별칭 반복('날카로운' 9회/3턴)으로 우회 → V9 반복 FAIL.
3. **v3 (최종)**: 주어 생략 기본 + **별칭 개시 문단 1회 제한** — 게이트 10/10 복귀.

**최종 구현**: 대화 턴(대상 NPC 존재) 상시 + 대명사 합산 키 감지 시, user 메시지 끝(근접 효과)에 hard 어투 디렉티브 — "대명사 개시 0회 / 주어 생략 기본 / 별칭 문단 1회" + 표시명 데이터(소개 여부 인지). `PRONOUN_OPENER_WHITELIST` 공용 export, soft 자제 라인에서 대명사 키 분리(이중 지시 희석 방지). 유닛 6케이스.

**실측** (12턴 게이트 2런, 모두 10/10 PASS·회귀 0):

| 조건 | 처방 전 | 처방 후 |
|---|---|---|
| 대화 턴 (chatty — 다중 NPC 순회) | 29~35% | **10.0%** (-66~71%) |
| 대화 턴 (devotee — 단일 NPC 장기 잠금, 최악) | ~29% | 28.1% (미미) |
| 합산 대화 턴 | ~29-35% | **18.8%** (상대 -40%) |
| 비대화 턴 | 3~8% | 2.7~5.0% (유지) |

**한계·잔여**: 단일 NPC 장기 잠금(devotee형)에서는 효과 미미 — 같은 인물이 매 턴 주어라 모델의 대명사 기본값이 디렉티브를 이긴다. 남은 수단은 ② 후처리 치환(문두 대명사→생략/별칭, 문장 파괴 위험)뿐이며 별칭 반복 풍선효과와의 상호작용 설계가 필요 — 착수는 소유자 결정 대기.
