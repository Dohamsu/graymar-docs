# 79. 프롬프트 토큰 최적화 — 측정 기반 스테이지별 예산 (2026-07-18)

> **상태**: 📐 계획 (P0 실측 완료, P1~P4 미착수)
> **동기**: 저렴한 모델(Gemma 4 26B **a4b** MoE — 활성 4B급 / gpt-4.1-nano) 운영 전제에서, 프롬프트가 길수록 지시 준수율이 떨어지는 한계를 **측정으로 정량화**하고, "지시를 잘 따르는 토큰량"을 스테이지별 예산으로 확정해 각각 최적화한다.
> **관련**: [[architecture/11_llm_prompt_caching|llm prompt caching]](설계만), arch/52(시스템 프롬프트 압축 -21%), 프롬프트 최적화 v2(선별 주입), arch/62(레이턴시), CLAUDE.md LLM 설계 원칙 6(프롬프트 최소주의).

## 1. P0 — 현황 실측 (2026-07-18, llm_call_logs 최근 3일 · 12턴×5런 포함)

| stage | 모델 | 호출 | avg 프롬프트 tok | max | avg 출력 | avg ms | 비용 비중 |
|---|---|---|---|---|---|---|---|
| **narrative** (메인 서술) | Gemma 4 26B | 1,588 | **10,856** | **15,429** | 220 | 4,418 | **61%** ($1.71) |
| narrative-retry | Gemma 4 26B | 262 | 10,085 | 14,469 | 192 | 4,857 | 7% |

### 발견 (실측 근거)

1. **메인 서술 프롬프트 = 명목 예산의 4.3배**. 불변식 17 "Token Budget 2500"은 실제로 **memoryParts 블록(assistant role 메모리)만** 트리밍한다 (prompt-builder L241~280). 시스템 프롬프트·facts 블록·디렉티브·세션 턴·NPC 블록은 무예산 — 총합 avg 10.9k / max 15.4k. 활성 4B급 모델에 15k 컨텍스트는 지시 희석이 유력한 구간.
2. **재시도율 16.5%** (narrative-retry 262/1,588) — 원인 미분류 (타임아웃/빈 응답/검증 실패). 재시도도 같은 10k 프롬프트로 나가 비용·레이턴시 이중 지불.
3. **intent 파서 비대**: 출력 37토큰(JSON 분류 1건)에 프롬프트 3,856토큰 — 시스템 프롬프트 단독 5,197자(예시 다수). 646호출(매 LOCATION 턴). 분류 정확도 대비 예시 과잉 여부 미측정.
4. **npc-reaction 2.4k** — nano 대상 프롬프트 중 최대. JSON 실패 재시도 이력 있던 스테이지(arch/67 — 실패 10.4%→재시도 구제).
5. **이상 신호**: narrative가 gpt-4.1-**nano**로 24건 실행됨 (fallback 정본은 4.1-mini) — 경로 미확인.
6. **캐시 미활용 의심**: cachedTokens 컬럼은 있으나 활용 집계 미확인. 시스템 프롬프트+정적 블록은 매 턴 동일한데 캐싱 설계(arch/11)는 미구현.

## 2. 목표

- **각 스테이지의 "준수율 유지 최대 토큰량"을 실측으로 결정**하고 스테이지별 예산으로 고정 (감이 아닌 곡선의 무릎에서 결정).
- 결과적으로: 메인 서술 프롬프트 총량 유예산화, 재시도율 하락, 턴 비용·레이턴시 절감 (부수 효과이지 1차 목표 아님 — 1차 목표는 **지시 준수율**).

## 3. 단계 계획

### P1 — 준수율 메트릭 정의 + 회고 상관 분석 (코드 0줄로 시작)

기존 자산으로 "프롬프트 크기 ↔ 지시 준수" 상관을 **이미 쌓인 데이터에서** 뽑는다:

- 데이터 조인: `llm_call_logs.calls[].promptTokens` × `turns.llm_prompt`(프롬프트 원문 저장됨) × 턴 서술 원문.
- **스테이지별 준수율 지표** (전부 기존 검증기 재사용):
  | stage | 준수 지표 (기존 자산) |
  |---|---|
  | narrative | 어체 준수율(llm_speech_audit), @마커 정합(V8), 금지어/메타 서술(audit_quality), 개시어율(D5-1), questReveal 반영, [최근 사용 표현] 준수 |
  | intent | actionType 정답률 (KW_OVERRIDE·수동 라벨 표본 ~100건) |
  | npc-reaction / nano-event | JSON 파싱 성공률·필드 유효율·재시도 발생률 (`attempts`, 서버 로그) |
  | dialogue | 어체 검증 통과율 (validateSpeechRegister 1차 통과 기준) |
- 산출: 스테이지별 **프롬프트 토큰 구간(2k 단위 버킷) × 위반율** 곡선. narrative는 표본 1,588건으로 즉시 가능.
- 재시도 원인 분류 (P0-2) + narrative@nano 24건 경로 규명 (P0-5).

### P2 — 예산 결정 (측정 → 숫자 고정)

- P1 곡선에서 위반율이 꺾이는 지점을 스테이지별 예산으로 채택. 곡선이 평평하면(크기 무상관) "현 크기 유지 + 비용만 절감" 모드로 전환 — **억지 삭감 금지** (규칙 희석 역효과, LLM 설계 원칙 6).
- 예산 후보 초안 (P1 결과로 대체될 가설값): narrative 총량 6~8k / intent 1.5k / npc-reaction 1.5k / nano-event 1.2k.
- 예산 상수는 `token-budget.service.ts`에 스테이지별로 외부화 (불변식 30 정신 — 하드코딩 금지).

### P3 — 스테이지별 최적화 실행 (우선순위 = 비용×준수 영향)

1. **narrative**: enforceTotal 스코프를 memoryParts → **전 블록**으로 확장 (시스템 프롬프트·고정 디렉티브는 보호 대상 명시, L0 테마 불변식 5 준수). 블록별 실측 기여도(P1에서 블록 단위 토큰 분해) 기반으로 LOW/HIGH 우선순위 재조정. 세션 턴 6개+THREAD 하이브리드의 중복 주입 여부 점검.
2. **intent**: 예시 축소 A/B — 정답률 표본(P1)으로 예시 절반/카테고리당 2개 버전 검증. 정확도 하락 시 원복 (정확도 > 토큰).
3. **npc-reaction / nano-event**: 주입 블록 선별 강화 (등장 NPC 외 정보 제거 — 선별 주입 불변식 24 재적용 점검).
4. **캐싱 (선택, P3b)**: OpenRouter/OpenAI prompt caching — 시스템 프롬프트+정적 블록을 메시지 선두 고정 배치로 캐시 적중 확보 (arch/11 설계 소생). cachedTokens 집계로 적중률 검증. 예산화와 독립적이라 병행 가능.
5. **재시도 경로**: P1 원인 분류 결과에 따라 — 검증 실패형이면 프롬프트 수정, 타임아웃형이면 arch/62 레버.

### P4 — 검증 게이트

- 각 최적화 단위마다: 유닛 green + **12턴×2런** (D5·V1~V10 전문항 + P1 준수율 지표 재측정) — 준수율 **비악화**가 통과 조건, 토큰·비용·레이턴시 절감은 보고 지표.
- 최종: 12턴×3런 누적 + NPA 감사 1회. 성공 기준 = ① 스테이지별 예산 내 p95 수렴 ② 준수율 전 지표 비악화 ③ 재시도율 16.5% → 10% 이하.

## 4. 비범위 (이번 트랙에서 하지 않음)

- 모델 교체/평가 (arch/25 별도 트랙), 스트리밍 구조 변경 (금지선 — arch/77 P4.0), JSON 모드 전환(스트리밍 비호환), 시스템 프롬프트 **내용** 재작성(압축은 arch/52에서 기완료 — 이번엔 배치·예산·주입 선별만).

## 5. 리스크

- **트리밍 = 동작 변경**: 블록이 잘리면 서술 품질이 미묘하게 변한다. 반드시 P4 게이트로 스텝별 확인 (arch/77 방식).
- 캐싱은 provider 의존 (OpenRouter 경유 Gemma 캐시 지원 여부 확인 선행).
- intent 예시 축소는 KW_OVERRIDE 안전망이 있으나 고위험 오분류(FIGHT 미탐 등)는 게이트에서 별도 확인.
