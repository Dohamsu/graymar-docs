# Node Resolve Rules Spec v1 (Canonical)

## 노드별 서버 처리 규약 (EVENT / REST / SHOP / EXIT / COMBAT)

------------------------------------------------------------------------

# 0. 목적

본 문서는 RUN/Node/Turn 상태머신 위에서, **각 노드 타입별 서버 처리
범위와 계약**을 정의한다.

-   "서버가 확정하는 것(SoT)" vs "LLM이 서술하는 것(표현)"을 분리
-   노드 진입/진행/종료 조건을 규격화
-   클라이언트가 LLM 없이도 UI를 복구/표시할 수 있게 최소 정보를 보장
-   DB/세부 스키마는 외부 플랫폼에서 확정 (본 문서는 스펙만)

------------------------------------------------------------------------

# 1. 공통 원칙

## 1.1 서버 SoT 우선

-   HP/스태미나/아이템/상태이상/거리/각도/골드/퀘스트 진행 등 **게임
    상태는 서버만 확정**
-   플레이어 입력이 "확정 결과"를 요구하더라도 서버는 **의도(intent)로만
    해석**하고 결과는 규칙으로 결정

## 1.2 server_result_v1는 "상태 변화 + 플레이어 가시 이벤트"만

-   내부 AI 판단/난수/디버그 로그는 server_result.events에 포함하지
    않는다
-   서술 텍스트는 LLM 의존(기본)이며 실패 시 폴백 서술로 대체 가능(운영
    정책)

## 1.3 Turn 단위 처리

-   모든 노드는 `/turns` 입력으로 진행한다 (freeform 또는 choice)
-   결과는 단일 트랜잭션으로 커밋 (turn/server_result/run/node_state)

## 1.4 멱등성/동시성

-   (runId, turnNo) 유니크 멱등
-   동일 turnNo 재요청 시 재실행 금지, 기존 결과 반환

------------------------------------------------------------------------

# 2. Node 타입 목록

-   COMBAT
-   EVENT
-   REST
-   SHOP
-   EXIT

> v1의 목표는 "한 RUN을 끝까지 굴리는 것"이며, 노드 타입은 추가
> 가능하다.

------------------------------------------------------------------------

# 3. Node 공통 인터페이스(개념)

각 노드는 다음 계약을 따른다.

## 3.1 NodeInput

-   `turnNo`
-   `input.type` ∈ { FREEFORM, CHOICE }
-   `input.text` 또는 `input.choiceId`

## 3.2 NodeState (노드별 상태)

-   nodeType마다 다르며, 반드시 `version`을 포함한다(권장)
-   복구를 위해 최소 표시 필드를 포함한다(예: SHOP 상품 목록, EVENT 단계
    등)

## 3.3 NodeOutput

-   `nextNodeState`
-   `server_result_v1`
-   `nodeOutcome` ∈ { ONGOING, NODE_ENDED, RUN_ENDED }

------------------------------------------------------------------------

# 4. COMBAT 노드 규약 (연결)

-   COMBAT는 별도 정본:
    -   `combat_resolve_engine_v1.md` (전투 수치 엔진)
    -   `status_effect_system_v1.md` (상태이상 시스템)
    -   `combat_engine_resolve_v1.md` §7 (구현 계약)
-   본 문서에서는 "노드 전이 규칙"만 명시

## 4.1 종료 조건

-   VICTORY / DEFEAT / FLEE_SUCCESS 중 하나면 `nodeOutcome = NODE_ENDED`
-   DEFEAT은 RUN 종료로 이어질 수 있음(런 정책에 따름)

## 4.2 COMBAT 종료 시 보상/정산

-   전리품/경험치/골드 등은 `rewards_and_progression_v1.md`에서
    정의(참조)

------------------------------------------------------------------------

# 5. EVENT 노드 규약

EVENT는 "서사/선택" 노드이며, v1에서 서버는 다음을 책임진다.

## 5.1 서버 처리 범위

-   이벤트 ID 선택/고정(진입 시)
-   선택지(choice) 생성(플레이어 가시)
-   선택 결과에 따른 **상태 변화**(HP/골드/아이템/퀘스트/평판 등)
-   이벤트 단계 진행 및 종료 조건

LLM은: - 이벤트 장면 묘사 - 선택지 문구의 표현(단, choiceId/효과는 서버
SoT) - 결과 서술(단, 상태 변화는 server_result를 기반으로)

## 5.2 EVENT NodeState(권장)

-   `eventId`
-   `stage` (정수 또는 ENUM)
-   `availableChoices[]` (id + label + optional hint)
-   `flags` (이미 선택했는지 등)

## 5.3 EVENT 처리 패턴(표준)

EVENT는 아래 3단계를 반복할 수 있다.

1)  INTRO: 상황 제시(선택지 제공 가능)
2)  CHOICE: 선택 입력 수신(CHOICE or FREEFORM→CHOICE 매핑)
3)  RESOLVE: 상태 변경 + 다음 stage 또는 종료

## 5.4 FREEFORM 입력 처리

-   EVENT는 FREEFORM을 직접 해석하지 않는다
-   서버는 FREEFORM을 **가장 가까운 choice로 TRANSFORM**하거나
-   매칭 실패 시 "다시 선택 요청" SYSTEM 이벤트를 반환한다

> **v1 구현 현황**: FREEFORM → CHOICE 매핑은 **미구현**. EVENT 노드에서는 CHOICE 입력만 처리한다. ACTION(FREEFORM) 입력은 COMBAT 노드에서만 지원된다.

## 5.5 EVENT 종료

-   stage가 END에 도달하면 `nodeOutcome = NODE_ENDED`
-   종료 시 `server_result`에 다음 노드로 이동 가능한 UI 힌트 포함(예:
    "계속" 버튼)

------------------------------------------------------------------------

# 6. REST 노드 규약

REST는 회복/정리 노드이다. v1은 단순 규칙을 고정한다.

## 6.1 서버 처리 범위

-   회복량 계산
-   상태이상 정리(선택적으로 일부만)
-   다음 노드로 전이

LLM은: - 휴식 장면 묘사 - 회복 결과 서술

## 6.2 REST 입력 형태

-   기본: CHOICE 기반
    -   "짧은 휴식"
    -   "깊은 휴식"
    -   "정찰/대화" (선택 사항)
-   FREEFORM은 가능한 한 CHOICE로 매핑

## 6.3 REST 표준 효과(권장 기본값)

-   Short Rest:
    -   stamina +1 (최대치 캡은 외부에서 정의)
    -   HP +10% maxHP
-   Long Rest:
    -   stamina +2
    -   HP +25% maxHP
    -   일부 상태이상 제거(예: DOT 1개 제거) --- 상세는 Status Spec
        확장로

> 수치는 외부 밸런싱에서 조정 가능. 본 스펙은 "형태"만 고정.

## 6.4 REST 종료

-   REST는 1\~2턴 내 종료를 원칙으로 한다
-   선택 수행 후 `nodeOutcome = NODE_ENDED`

------------------------------------------------------------------------

# 7. SHOP 노드 규약

SHOP은 거래 노드이다. v1에서는 "구매 중심"으로 단순화 가능.

## 7.1 서버 처리 범위

-   상품 리스트 확정(진입 시)
-   가격/재고 확정(SoT)
-   구매 처리(골드 감소, 인벤 증가)
-   판매 처리(선택 사항)
-   종료/이탈

LLM은: - 상점 장면/상인 대사 - 상품 설명(단, 가격/효과는 서버 기반)

## 7.2 SHOP NodeState(권장)

-   `shopId`
-   `catalog[]` (상품ID, 표시명, 가격, 재고, 간단 설명)
-   `playerGoldSnapshot` (표시용)
-   `ui` (장바구니형이 필요하면 별도)

## 7.3 입력 처리(표준)

-   CHOICE 또는 COMMAND형 FREEFORM ("포션 2개 구매")
-   서버는 구매 가능한 상품/수량을 검증하고:
    -   가능: PURCHASE 이벤트 + 상태 변경
    -   불가: SYSTEM 이벤트(사유) + 상태 변화 없음

## 7.4 SHOP 종료

-   "나간다" 선택 시 NODE_ENDED
-   catalog는 노드가 유지되는 동안 고정(권장)

------------------------------------------------------------------------

# 8. EXIT 노드 규약

EXIT는 노드 종료가 아니라 "RUN 종료"를 표현할 수 있다.

## 8.1 서버 처리 범위

-   RUN 정산 트리거(필요 시)
-   영구/임시 보상 반영(별도 스펙 참조)
-   RUN 상태를 ENDED로 전환

LLM은: - 엔딩/정산 서술

## 8.2 EXIT 입력 형태

-   보통 CHOICE:
    -   "귀환한다(종료)"
    -   "더 진행한다(취소)"(선택)

## 8.3 종료

-   귀환 선택 시 `nodeOutcome = RUN_ENDED`
-   run.status = ENDED

------------------------------------------------------------------------

# 9. 노드 전이(Transition) 공통 규칙

## 9.1 nextNode 선택

-   다음 노드는 "노드 생성기/플래너"가 결정한다(별도 스펙)
-   본 스펙에서는 다음만 보장:
    -   `server_result.ui.nextNodeHint` 제공 가능
    -   run.currentNodeId 업데이트는 트랜잭션 커밋에 포함

## 9.2 노드 진입 시스템 메시지

노드 진입 시 SYSTEM kind 이벤트로 자연스러운 한국어 메시지를 생성한다.

| nodeType | 진입 메시지 |
|----------|------------|
| COMBAT | 전투가 시작된다! |
| EVENT | 새로운 상황이 펼쳐진다. |
| REST | 휴식을 취할 수 있는 장소에 도착했다. |
| SHOP | 상점에 도착했다. |
| EXIT | 여정의 끝이 보인다. |
| (기타) | 다음 구간으로 이동했다. |

## 9.3 전이 시 컨텍스트 전달

-   노드 종료 시, 다음 노드에 전달할 최소 컨텍스트:
    -   전투 결과(승/패/도주)
    -   획득 아이템/골드 변화
    -   퀘스트 플래그 변화
    -   주요 이벤트 로그 요약(LLM 컨텍스트용)

## 9.4 클라이언트 이벤트 필터링

> **v1 구현 현황**: 기본적으로 클라이언트는 `SYSTEM`, `LOOT`, `GOLD` kind 이벤트만 유저에게 표시한다. 단, **전투 LLM 스킵 시**(llm.status === 'SKIPPED') `BATTLE`, `DAMAGE`, `MOVE`, `STATUS` 이벤트도 SYSTEM 메시지로 유저에게 표시한다. 전투 중 턴은 `options: { skipLlm: true }`로 제출되어 LLM 내러티브를 생략하고 1-2초 내 결과를 반환한다. 전투 진입/종료 시에는 LLM 내러티브가 정상 생성된다.

> 컨텍스트 캐싱/요약은 별도 LLM Context Spec에서 정의

------------------------------------------------------------------------

# 10. UI 최소 힌트(권장)

각 server_result에는 클라가 UI를 구성할 수 있도록 최소 힌트를 포함할 수
있다.

-   `ui.availableActions` (현재 노드에서 가능한 입력 타입)
-   `ui.choices[]` (있는 경우)
-   `ui.primaryCTA` ("계속", "구매", "휴식" 등)
-   `ui.blockingReason` (입력 거부 사유)

------------------------------------------------------------------------

# 11. 테스트 시나리오(텍스트)

-   EVENT: choice 선택 → 퀘스트 플래그 변경 → NODE_ENDED
-   REST: long rest 선택 → HP/스태미나 회복 → DOT 1개 제거 → NODE_ENDED
-   SHOP: 골드 부족 구매 시도 → SYSTEM 이벤트 → 상태 변화 없음
-   EXIT: 귀환 선택 → RUN_ENDED → 이후 /runs/{runId}에서 종료 상태 확인
-   FREEFORM: EVENT에서 애매 입력 → 선택지 재제시

------------------------------------------------------------------------

# 결론

본 문서는 v1에서 RUN이 "전투만이 아니라" EVENT/REST/SHOP/EXIT까지 포함해
완주 가능하도록 만드는 노드별 서버 처리 규약(정본)이다.
